  - 조정대상지역 다주택자 양도 시 기본세율 + 20%p(2주택) / +30%p(3주택) 적용

- **영향 범위**: tax-rules.js, scenario 가정값, 입력 검증

### #2. 개발 범위와 마일스톤

- **결정일**: 2026-04-26
- **결정**: A안 — 5/7 2차 발표까지 v0.1 → v0.2 → v0.3 단계적 진행 (12일 일정).
- **이유**: PRD 5-1절 P0 핵심 메시지("순서·시점 비교로 세후 실수령액 차이 발생")를 데모에서 살리기 위해서는 단일 주택 일반과세만으로 부족하며, 비과세·중과·시나리오 엔진까지 필요함. v0.1만으로는 두 시나리오 결과가 같게 나와 데모 메시지 전달 불가.
- **버전별 범위**:
  | 버전 | 범위 | 목표일 |
  |---|---|---|
  | v0.1 | 단일 주택 일반과세 계산 | 4/28(화) |
  | v0.2 | + 장특공 + 1세대1주택 비과세 | 5/1(금) |
  | v0.3 | + 다주택 중과 + 시나리오 엔진 + 상태전이 | 5/4(월) |
  | v0.4 | + 결과 화면 동적 렌더링 + 설명 문장 | 5/6(수) |
  | 발표 | 2차 발표 | 5/7(목) |
- **후퇴 시나리오**: 5/4 시점에 v0.3가 동작하지 않으면 즉시 v0.2로 후퇴. 5/2 시점에 시나리오 엔진 상태전이가 막히면 시나리오를 2개로 고정(A→B / B→A)하고 3주택 순열 확장은 5/7 이후로 미룸.
- **영향 범위**: 전체 일정, 모듈 구현 순서

### #3. salePlan UI 통합 위치

- **결정일**: 2026-04-26
- **결정**: 기존 ②세대 기본 정보 카드와 ③주택별 정보 입력 카드 사이에 "②.5 양도 계획" 카드를 신설한다.

- **영향 범위**: index.html, input-collector.js

- **결정**: 자연어 입력 → 폼 자동 채우기 기능을 5/7 데모 범위에서 제외한다.
- **이유**: PRD P0 핵심 가치(입력 → 조합 생성 → 세액 계산 → 최적 추천)에 포함되지 않으며, KPI에도 무관. 발표 시연 중 LLM이 엉뚱한 값을 뽑을 사고 위험 존재. 계산 엔진 일정에 자원 집중이 우선.
- **세부 처리**:
  - 기존 nlp-area 텍스트영역 숨김 또는 제거
  - btn-ai-extract 버튼 숨김 또는 제거
  - dummyExtract(), runAiExtract() 함수는 코드 남겨두되 호출되지 않도록
  - highlightMissing(), field-error 빨간 테두리, missing-banner는 수동 입력 검증에도 유용하므로 **유지**
- **재검토 시점**: 5/7 이후 v0.5 단계에서 검토
- **영향 범위**: index.html

### #5. 코드 파일 구조

- **결정일**: 2026-04-26
- **결정**: 계산·시나리오 관련 코드는 인라인 `<script>`가 아닌 별도 .js 파일로 분리. ES6 module은 사용하지 않고 비-모듈 `<script src="...">` 다중 로드 방식으로 한다.
- **이유**:
  - v0.3까지 가면 코드량이 600~800줄로 폭증해 단일 HTML로는 관리 한계
  - PRD 11절 KPI "규칙 테스트 통과율 95% 이상" 충족을 위해 함수가 import 가능해야 함
  - index.html과 result.html이 같은 함수를 공유하려면 별도 파일이 필수
  - 시스템 프롬프트의 "화면과 계산 로직 분리 원칙"과 일치
  - ES6 module 대신 비-모듈 방식을 택한 이유: 로컬 file:// 환경과 GitHub Pages 모두 추가 도구 없이 동작
- **권장 디렉토리 구조**: 의사결정 #7 참조
- **PRD와의 관계**: PRD 8-8 "계산엔진: JavaScript(클라이언트 사이드)"는 "어디서 돌릴 것인가" 결정. 본 의사결정은 "어떻게 모듈화할 것인가" 결정으로 PRD와 충돌하지 않음.
- **영향 범위**: 모든 코드 파일 구조, Codex 작업지시서
| 2026-04-27 (v3) | #5 보강: 파일명 언더바(snake_case) 명명 규칙 추가. 디렉토리 구조 예시 갱신 |
- **파일명 명명 규칙** (2026-04-27 v3 추가):
  - 모든 .js 파일명은 **언더바(snake_case)** 사용
  - 예: `tax_rules.js`, `tax_engine.js`, `scenario_engine.js`, `input_collector.js`, `result_renderer.js`, `explanation_engine.js`
  - 테스트 파일도 동일: `tests/test_cases.js`
  - 이유: Python 관행 + GitHub URL 가독성 + 일관성
